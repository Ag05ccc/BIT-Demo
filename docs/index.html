Product Test BIT

Built-In Test (BIT) is a client-server diagnostics application for robotics and drone systems. Test logic runs on the target device (Jetson, embedded Linux) with direct access to autopilot, peripherals, and ROS. A Test PC triggers tests and displays live results via browser or CLI — no hardware knowledge required on the operator side.

Python 3.8+ Flask REST API No JS frameworks Stdlib-only client ROS / MAVROS MIT License

Quick Start

Option 1 — Local Simulation (no hardware)

Run the entire stack on your PC — no Jetson, ROS, or peripherals required.

# Install server dependencies
pip install flask psutil

# Start sim server + open browser at http://localhost:5500
python3 run_local.py

# Or: start server + immediately run all tests via CLI
python3 run_local.py --run

# Or: run a single category
python3 run_local.py --run autopilot

Option 2 — SSH Launcher (recommended for real hardware)

1

One-time SSH key setup on your PC:

ssh-copy-id ubuntu@192.168.1.2
2

Start the launcher on your PC:

pip install flask
python3 launcher.py       # opens http://localhost:8080
3

Enter the Jetson IP, click CONNECT. The launcher SSHes in, starts test_server.py, then opens the BIT dashboard at http://<jetson-ip>:5500 automatically.

Option 3 — Manual SSH

# Copy project to Jetson (first time only)
scp -r BIT-Demo ubuntu@192.168.1.2:~/BIT-Demo

# SSH in and start server
ssh ubuntu@192.168.1.2
cd ~/BIT-Demo/server
pip3 install flask psutil pyserial
python3 test_server.py

Open http://192.168.1.2:5500 in your browser.

Server Run Modes

Production — real hardware

cd ~/BIT-Demo/server
python3 test_server.py

Simulation — no hardware

Replaces all real checks with sim_checks.py. Pass/fail rates and specific overrides are controlled by config_local.json.

python3 test_server.py --sim
Sim config keyTypeDescription
pass_ratefloat 0–1Probability a check passes
warning_ratefloat 0–1Probability of warning from the non-pass pool
random_seedint | nullSet for reproducible results; null for random
overridesobjectForce specific checks to a fixed status: {"MetadataCaptureCheck": "passed"}

Debug mode

Attaches the source file and line number of every error/warning to the check result, and includes full Python tracebacks. Both the Web GUI and CLI display this extra information automatically.

python3 test_server.py --debug          # production + debug
python3 test_server.py --sim --debug    # simulation + debug

Keep running after SSH disconnect

# nohup (simple)
nohup python3 test_server.py > server.log 2>&1 &

# screen (re-attachable)
screen -S bit
python3 test_server.py
# Detach: Ctrl+A, D   |   Re-attach: screen -r bit

CLI Client

Runs on the Test PC. No external dependencies — uses Python stdlib only (urllib, json, argparse).

# Configure Jetson IP
nano client/client_config.json

# Local testing (server on same machine)
python3 client/test_client.py --config client/client_config_local.json run

run — Start tests

python3 client/test_client.py run              # all categories
python3 client/test_client.py run autopilot    # one category
python3 client/test_client.py run ros
python3 client/test_client.py run network

Results stream live to the terminal as each check completes. A summary and solution suggestions appear at the end.

status — Server info

python3 client/test_client.py status

Shows hostname, OS, CPU, RAM, disk of the target device.

results — Latest results

python3 client/test_client.py results

Displays the most recent test run without triggering a new one.

report — Download HTML report

python3 client/test_client.py report

Downloads a self-contained HTML report and saves it locally as bit_report_<test_id>.html.

export-params / compare-params

python3 client/test_client.py export-params    # export autopilot params
python3 client/test_client.py compare-params   # compare with defaults

Custom config path

python3 client/test_client.py --config /path/to/my_config.json run

SSH Launcher

launcher.py is a small Flask web app (port 8080) that runs on your PC and provides a graphical entry point to start the BIT server on the device via SSH.

# One-time SSH key setup
ssh-copy-id ubuntu@192.168.1.2

# Start launcher (opens http://localhost:8080)
python3 launcher.py
✓

If the BIT server is already running on the device, the launcher page detects it on load and shows an Open Dashboard button immediately — no SSH needed.

How it works

1

Enter device IP and click CONNECT.

2

Launcher SSHes in (BatchMode=yes — key auth required), kills any existing server instance, and starts a fresh one with nohup python3 test_server.py > /tmp/bit_server.log 2>&1 &.

3

Polls /api/status up to 30 s until the server responds.

4

Auto-opens the BIT dashboard at http://<device-ip>:5500 in a new browser tab.

Scripting API

Use APIClient directly for CI/CD pipelines or custom automation:

import sys, os, time
sys.path.insert(0, 'path/to/BIT-Demo')
from client.utils.api_client import APIClient

client = APIClient('http://192.168.1.2:5500')

# Start tests
resp    = client.run_tests()          # or run_tests(category='ros')
test_id = resp['test_id']

# Poll until complete
while True:
    results = client.get_results(test_id)
    if results['status'] == 'completed':
        break
    time.sleep(1)

# Check outcome
summary = results['summary']
print(f"Passed: {summary['passed']}/{summary['total']}")

for r in results['results']:
    if r['status'] == 'failed':
        print(f"FAIL: {r['name']} — {r['message']}")

# Download HTML report
report_bytes = client.get_report(test_id)
with open(f"report_{test_id}.html", 'wb') as f:
    f.write(report_bytes)

Troubleshooting

ProblemFix
Cannot connect to serverVerify server is running; check IP in client_config.json; ping device
Port refusedDefault port is 5500 — make sure config matches; check firewall rules
SSH CONNECT failsRun ssh-copy-id ubuntu@<ip> first — key auth is required (BatchMode=yes)
Port already in usesudo lsof -i :5500 → sudo kill -9 <PID>
Permission denied for serial devicessudo usermod -a -G dialout ubuntu and log out/in
ROS checks skippedSource ROS: source /opt/ros/noetic/setup.bash; ensure ROS_MASTER_URI is set in config
Autopilot checks failingVerify MAVROS: rosnode list | grep mavros; check mavros_ns in config; increase heartbeat_timeout
Slow response / timeoutIncrease timeout in client config; some checks (ROS rate measurement) take several seconds by design
Results not updatingOnly one test can run at a time — wait for the current run to finish
